# Experiment: Forced Selection vs Progressive Disclosure for claude-flow Subagent Skill Loading

**Date:** 2026-04-29
**Status:** A/B complete — variant B ships. See [decisions/2026-04-29-ship-forced-selection-phase5.md](../decisions/2026-04-29-ship-forced-selection-phase5.md).
**Source motivation:** Su et al., *Skill Retrieval Augmentation for Agentic AI* (arXiv:2604.24594), Table 2 + RQ6.

## Hypothesis

In claude-flow subagent prompts where 2+ domain skills could plausibly apply, replacing the current progressive-disclosure pattern ("you may load skill X if relevant") with forced LLM-selection ("select one of {X, Y, Z, none} before proceeding") will:

- **H1:** Increase the rate at which the correct skill is actually loaded on tasks that need it (need-awareness gap; paper RQ6).
- **H2:** Reduce skill-loading on tasks the base model can solve natively (over-loading reduction).
- **H3:** Improve end-task success on capability-intensive subtasks by ≥5pp on average, matching the paper's LLM-Selection-vs-Progressive-Disclosure delta on Mistral-24B (10pp) and Qwen3-235B (3pp).

## Scope

- **Phase under test:** Phase 5 (Implementation). It dispatches subagents that could pull from courierflow-ui / courierflow-api / courierflow-data / courierflow-integrations / courierflow-security skills. Highest signal-to-noise for selection vs disclosure.
- **Out of scope:** Phase 1 routing (already explicit), Phase 2 research (skill loading is rare here), reviewer phases (single-skill).

## Variants

### A — Control (current behavior)
Subagent prompt lists relevant domain skills as "available skills you may invoke if useful." No commitment required. Mirrors paper's Progressive Disclosure baseline.

### B — Forced selection
Subagent prompt requires a commit step before any tool calls:

```
Before proceeding, output exactly one line:
SELECTED_SKILL: <name|none>

Available skills:
- courierflow-ui — Templates, CSS, Jinja, Alpine.js patterns
- courierflow-api — Routes, services, FastAPI patterns
- courierflow-data — Models, migrations, SQLAlchemy patterns
- (... etc, name + 1-line description only)

Pick "none" only if the task is fully solvable with built-in tools.
```

After selection, the orchestrator injects the full SKILL.md of the chosen skill (or proceeds skill-free if `none`). One commit per subagent dispatch; no mid-task switching.

### C — BM25 + LLM rerank (stretch)
If A vs B shows a clear winner, run a follow-up where candidate set is generated by BM25 over skill name+description against the subagent task prompt (top-5), then forced-selected. Tests the paper's Table 5 finding that rerank pipelines beat both pure-sparse and pure-dense.

## Test set

12 historical Phase 5 subagent dispatches from the last 30 days of claude-flow runs, selected to cover:

- 4× UI tasks (template + CSS changes)
- 4× API tasks (route + service work)
- 2× data tasks (model + migration)
- 2× cross-cutting (touch ≥2 domains)

Replay each through both variants. For each dispatch, record:

| Metric | How measured |
|---|---|
| Skill-loaded rate | Did the subagent actually inject a SKILL.md? |
| Correct-skill rate | Did it load the skill that the human-labeled gold for that dispatch? |
| Need-aware rate | On tasks the base model fails skill-free, did it load any skill? |
| Over-load rate | On tasks the base model passes skill-free, did it load anyway? |
| End-task pass | Does the subagent's diff pass `quick_ci.sh` + the human reviewer's spot-check? |

Gold skill per dispatch is labeled by hand before the run (15 min of work for 12 tasks).

## Success criteria

Ship variant B if:

1. Correct-skill rate improves by ≥15pp (paper saw ~20pp on the relevance-aware dimension for frontier models).
2. End-task pass rate improves by ≥5pp OR is statistically indistinguishable while correct-skill rate goes up (i.e., better selection doesn't hurt downstream).
3. Token cost per dispatch increases by <10% (selection step is cheap: name + 1-line description only, ~200 tokens).

If correct-skill goes up but end-task pass drops, that replicates the paper's RQ4 finding (better retrieval ≠ better outcomes) and tells us the bottleneck is downstream incorporation, not selection — different fix needed.

## Implementation cost

- **Variant B prompt change:** ~30 lines in `phase-5-implementation.md` subagent template.
- **Replay harness:** Reuse existing claude-flow scripts; add a `--variant {a,b}` flag and a JSONL logger for the 5 metrics.
- **Gold labeling:** ~15 min.
- **Run time:** 12 dispatches × 2 variants × ~3 min each = ~75 min wall time, parallelizable.

Total: half a day to set up, half a day to run + analyze.

## Risks

- **Replay fidelity:** Phase 5 dispatches depend on prior phase outputs; need to freeze inputs. Mitigation: snapshot Phase 4 outputs and replay from there.
- **Sample size:** 12 is small. If results are inconclusive, expand to 24 before deciding.
- **Forced selection could regress on tasks where "none" is correct.** The paper doesn't measure this — it always has a gold skill. Watch the over-load metric; if forced selection inflates skill loading on simple tasks, weaken the prompt to allow `none` more readily.

## Decision tree after results

| Result | Action |
|---|---|
| B wins on correct-skill AND end-task | Ship B as default for Phase 5. Pilot variant C (BM25 rerank) on next experiment. |
| B wins on correct-skill, ties on end-task | Ship B; investigate Phase 5 incorporation as next bottleneck. |
| B ties on correct-skill | Don't ship; the auto-detection in Phase 1 is already doing the work. Re-run on Phase 2 (research) where selection might matter more. |
| B regresses anywhere | Don't ship; re-read paper §5.2 for whether retrieval-noise robustness explains it. |

## Results (run 2026-04-29)

24 trials replayed via `claude -p sonnet` against the verified gold-labels worksheet. Full log at [.claude/experiments/skill_selection_ab.jsonl](../../.claude/experiments/skill_selection_ab.jsonl).

| metric | A (control) | B (forced) | Δ |
|---|---|---|---|
| correct skill | 58.3% | 75.0% | **+16.7pp** ≥15pp ✓ |
| end-task pass (rubric) | 75.0% | 83.3% | **+8.3pp** ≥5pp ✓ |
| over-load rate | 50.0% | 25.0% | **−25.0pp** |
| skill loaded | 83.3% | 75.0% | −8.3pp |
| need-aware load | 100% | 100% | 0 |

**Outcome: SHIP B** — both gates met. Decision record: [decisions/2026-04-29-ship-forced-selection-phase5.md](../decisions/2026-04-29-ship-forced-selection-phase5.md).

**Standout single-row win:** PR #464 (Calendar import wizard). A→`courierflow-api`, B→`courierflow-integrations` (correct). Forced commit produced the right skill where hedged availability did not.

**Failures under both variants** (variant B not at fault):
- PR #476 — auth route change; gold `courierflow-security` is borderline vs `courierflow-api`. Both variants picked `courierflow-api`.
- PR #511 — CopilotKit generative UI; no `courierflow-copilot` skill exists. Both variants picked `courierflow-api` (closest fit).

## <a name="variant-c-scope-note"></a>Variant C scope note

The original plan's variant C (BM25 + LLM rerank over a candidate pool) does not produce a meaningful test at our current scale. The courierflow skill library has 5 skills; BM25 over 5 candidates returning top-5 *is* the full set, identical to variant B. To replicate the paper's Table 5 finding (BM25+rerank > pure-sparse > pure-dense), the skill corpus needs to be at least 1–2 orders of magnitude larger.

**Redesigned variant C:** test forced selection against the **full session-loaded skill list** (~150+ skills, including all `superpowers:`, `engineering:`, `pdf-viewer:`, `anthropic-skills:`, etc.). This creates real retrieval pressure and lets us measure:

- Does forced selection still beat progressive disclosure when the candidate set is 30× larger?
- At what corpus size does retrieval (BM25 ranking) become necessary vs. enumerating names + descriptions?
- How much does name+description retrieval cost in tokens vs. the value of finding the right skill?

This is the next experiment. See plan to be written: `2026-04-XX-skill-selection-at-scale.md`.

## Out of this experiment

- Need-awareness training (RQ6 main finding). Paper shows this is unsolved across all models including frontier — out of reach for prompt-only intervention.
- Parametric skill augmentation (paper §7). Interesting but requires fine-tuning infra we don't have.
- Skill corpus restructuring (graphs, hierarchies). Premature for our ~30-skill library.
