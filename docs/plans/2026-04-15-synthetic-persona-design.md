# Synthetic Persona Skill — Design Document

**Date**: 2026-04-15
**Status**: Approved
**Skill name**: `synthetic-persona`

## Concept

Build a "synthetic person" from public data, then use that persona to review your product, brainstorm ideas, rehearse conversations, or analyze interpersonal dynamics — getting work to 80-90% before the real conversation.

Inspired by the "synthetic client" and "What Would [Boss] Do" workflows: gather public information about a real person, have AI generate its own persona instructions from that material, then role-play as that person for product feedback, sounding-board sessions, or pitch rehearsals.

## Principles

1. **Public data only** — never use private or confidential information. The skill explicitly warns against this.
2. **Meta-prompted persona** — Claude generates its own role-playing instructions from source material, producing richer behavior than a fixed template.
3. **Project-agnostic** — works on any codebase or product, not hardcoded to a specific project.
4. **Pre-vetting, not replacement** — framed as getting to 80-90%, not substituting for real human feedback.
5. **Confidence disclosure** — reports flag where the persona model is thin and what to validate with the real person.
6. **Personas are reusable** — saved to memory for future sessions.

## Stages

### Stage 1: Research & Build Persona

**Goal**: Collect public data about a real person and generate a persona that Claude can role-play.

**Entry question**: "Who would you like to model? Tell me their name, role, and relationship to you/your product."

**Data collection — source categories:**
- Articles / blog posts they've written
- Podcast or interview appearances
- Public talks, keynotes, presentations
- Operating manuals / "how I work" docs (if shared publicly or provided by user)
- Social media posts showing priorities, opinions, frustrations
- Framework data (Enneagram type, DISC profile, management style — if known)
- Industry context (what market they operate in, what problems they face daily)

**Process:**
1. Ask the user what sources they have or can point to
2. Use web search to find additional public material
3. For each source, extract: priorities, communication style, decision patterns, domain expertise, frustrations, values
4. Synthesize into a **Persona Card** (structured reference for the human)
5. **Meta-prompt step**: Feed all gathered material to Claude and ask it to generate a **Persona Prompt** — the actual instructions for role-playing this person. This produces richer, more nuanced behavior than a fixed template.

**Persona Card output:**
```markdown
## Persona — [Name]

### Identity
[Role, title, relationship to user/product]

### Priorities
[What they care about most — ranked]

### Communication Style
[Direct/diplomatic, data-driven/intuitive, formal/casual, verbose/terse]

### Domain Expertise
[What they know deeply, what they're less familiar with]

### Tech Comfort
[Power user / competent / casual / reluctant]

### Known Frustrations
[Pain points they've expressed publicly]

### Decision Pattern
[How they evaluate — ROI-focused? User-first? Risk-averse? Data-driven? Gut-feel?]

### Personality Frameworks
[Enneagram, DISC, MBTI, etc. — if known]

### Jobs-to-Be-Done
Primary: When [situation], I want to [motivation], so I can [expected outcome].
- Functional: [practical task to accomplish]
- Emotional: [how they want to feel]
- Social: [how they want to be perceived]

### Empathy Map
- **Says**: [direct quotes from public sources]
- **Thinks**: [worries, aspirations inferred from content]
- **Does**: [observable behaviors, workarounds]
- **Feels**: [emotional patterns — frustrations, delights]

### Sources
[Links to public materials used to build this persona]
```

**Persona Prompt**: A separate block of instructions generated by Claude from the source material — written in second person ("You are [Name]. You think about problems by..."). This is what Claude actually uses when adopting the persona in Stage 3.

**Save both to memory** for reuse in future sessions.

### Stage 2: Product / Context Scan

**Goal**: Read the current project to build a "product map" the persona will review.

**What to scan:**
- UI templates (HTML, components) — user-facing copy, labels, flows
- Routes / API endpoints — what the product can do
- Error messages, validation messages — how failures are communicated
- README, docs — how the product presents itself
- Feature set — what exists, what's in progress (from git history / TODOs)
- Onboarding flows — first-time user experience

**Output**: A concise product summary organized by user-facing areas (not code architecture). Example sections: "Getting Started", "Core Workflow", "Settings & Configuration", "Notifications", "Error Handling".

**Optional journey map**: For end-to-end reviews, map the user experience across stages (Awareness → Onboarding → Core Use → Advanced Use → Retention). Each stage tracks touchpoints, actions, emotions, pain points, and JTBD alignment. More structured than the feature-by-feature product map.

**Two-pass enrichment**: After building the product map, feed a summary back into the persona: "Now that you know what [Product] does, refine your perspective on what matters most to [Name] about this product." This tightens reactions from generic to product-specific.

### Stage 3: Interactive Session

**Choose a mode at session start:**

#### Mode A: Product Review (Synthetic Client)

Claude adopts the persona and walks through the product map feature-by-feature.

- Reacts as the person would: "As [Name], I'd be confused by...", "This solves my [priority] directly", "Where's the [expected feature]?"
- User can present specific features, mockups, or proposals
- Back-and-forth dialogue — user can push back, clarify, redirect
- Persona stays in character but flags when it's speculating beyond source data

**Prompt pattern**: "Review [feature area] from [Name]'s perspective. What would they approve of? What would confuse them? What would they expect that isn't here?"

#### Mode B: Brainstorming (Sounding Board)

Claude adopts the persona for open-ended problem-solving.

- User presents a challenge, idea, or decision
- Persona responds with their characteristic thinking style
- Good for: "How would [Name] approach this?", "What would [Name] push back on?", "What questions would [Name] ask?"
- Functions as an on-demand co-founder / advisor / stakeholder

**Prompt pattern**: "I'm considering [idea/decision]. React as [Name] would — what questions would you ask? What concerns would you raise? What would excite you?"

#### Mode C: Rehearsal (Pitch Practice)

Claude adopts the persona so the user can practice high-stakes conversations.

- User pitches an idea, asks for resources, proposes a change
- Persona reacts as the real person would — pushback, questions, approval signals
- Helps identify objections before the real conversation
- After the rehearsal, Claude breaks character to debrief: "Here's what landed well, here's what you should tighten"

**Prompt pattern**: "I'm going to pitch you [idea]. React as [Name] would — ask the questions they'd ask, raise the concerns they'd raise. After we're done, break character and tell me how to improve."

#### Mode D: Dynamics Analysis (Optional)

Requires: the user's own personality profile (from personal-coach or provided directly) AND the synthetic persona.

- Analyzes the dynamic between the two profiles
- Identifies natural alignment points and friction areas
- Suggests communication strategies tailored to the pairing
- Example: "Your Enneagram 9 tendency to avoid conflict meets [Name]'s Type 5 need for directness — lead with data, not consensus"

**Prompt pattern**: "Given my profile [summary] and [Name]'s persona, how should I approach [topic]? Where will we naturally align? Where should I adjust my style?"

### Stage 4: Findings Report

**Goal**: Synthesize the interactive session into a structured, actionable report.

**Output format:**
```markdown
## Persona Review — [Name] on [Project]
### Date: [date]
### Mode: [Product Review / Brainstorming / Rehearsal / Dynamics]
### Persona confidence: [High/Medium/Low — based on depth of source material]

---

### What Works
[Features, approaches, or ideas the persona approved of, with reasoning tied to their known priorities]

### What's Confusing
[Terminology, flows, or concepts that don't match this person's mental model]

### What's Missing
[Features, information, or capabilities the persona would expect to exist]

### Priority Issues (Opportunity Scored)
Ranked using Opportunity Score = Importance + (Importance - Satisfaction). Scores > 10 = high-priority gaps.

| Issue | Importance (1-10) | Satisfaction (1-10) | Opp. Score | Action |
|-------|-------------------|---------------------|------------|--------|
| [gap] | [how much persona cares] | [how well product addresses it] | [calculated] | [fix/add/improve] |

### Key Reactions
[Notable quotes or reactions from the interactive session worth preserving]

### Confidence & Caveats
[Where the persona model is strong vs. thin. What conclusions to validate with the real person. Areas where Claude was speculating beyond source data.]

### Recommended Next Steps
[Concrete actions — what to refine before the real conversation]
```

**Saved to**: `docs/persona-reviews/YYYY-MM-DD-[name]-review.md`

## What This Skill Does NOT Do

- **Replace real human feedback** — it's pre-vetting, not a substitute
- **Use private/confidential information** — public sources only
- **Operationalize sales frameworks** — that's a separate skill concern
- **Provide therapy or mental health support** — if dynamics analysis surfaces personal distress, recommend professional help
- **Guarantee accuracy** — the confidence section exists because personas are approximations

## Relationship to Other Skills

- **personal-coach**: Provides the user's own personality profile for Stage 3D (Dynamics Analysis). The two skills are complementary — personal-coach for self, synthetic-persona for others.
- **brainstorming**: Synthetic-persona's Mode B overlaps with general brainstorming but adds persona grounding. Use synthetic-persona when you want a specific person's perspective; use brainstorming for general creative exploration.
- **debate-team**: Could be composed with synthetic-persona — have multiple personas review a proposal from different angles. Future enhancement.
